# Задача 04 — `git push` и `git pull`

## Новые команды

```bash
git push origin <ветка>   # отправить ветку на GitHub
git pull                   # получить изменения с GitHub
git fetch                  # получить, но не применять
```

`push` — отправляет ваши локальные коммиты на удалённый репозиторий.  
`pull` — забирает изменения оттуда и применяет к вашему локальному коду.  
Это основа совместной работы: так несколько человек работают над одним проектом.

---

## Физическая модель

Представьте: вы работаете над симуляцией на домашнем компьютере,
а я хочу посмотреть результат и оставить комментарий.
Без `push` ваши изменения существуют только у вас — они невидимы для остальных.

---

## Задание

### Шаг 1 — Опубликовать ветку

Отправьте ветку `feature/initial-conditions` на GitHub:

```bash
git push origin feature/initial-conditions
```

Зайдите на GitHub — убедитесь, что ветка появилась в списке веток репозитория.

### Шаг 2 — Смоделировать правку преподавателя

Откройте файл `README.md` прямо в браузере на GitHub и добавьте одну строку —
например, описание параметров функции `string()`. Сохраните через кнопку Commit.

### Шаг 3 — Получить изменения локально

```bash
git checkout main
git pull
git log --oneline
```

Убедитесь, что правка из браузера появилась в локальной истории.

### Шаг 4 — Разница между `fetch` и `pull`

```bash
git fetch origin
git diff main origin/main
```

`fetch` только скачивает информацию об изменениях, но не применяет их.
`diff` покажет разницу между вашей локальной веткой и тем, что на GitHub.

---

## Подсказки

- Если `git push` отклонён с ошибкой `rejected` — сначала сделайте `git pull`, затем снова `push`
- `origin` — это имя удалённого репозитория (GitHub). Его можно проверить через `git remote -v`
- `git fetch` безопасен: он никогда не меняет ваши файлы

---

## Ожидаемый результат

- Ветка `feature/initial-conditions` видна на GitHub
- Локальная история содержит правки, сделанные через браузер GitHub
- Вы понимаете разницу между `fetch` и `pull`
